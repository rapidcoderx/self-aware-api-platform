## The One-Line Answer

Voyage AI `voyage-4` delivers the best semantic retrieval accuracy at 1024 dimensions,
is purpose-built for the technical domains our API schemas live in (finance and code),
requires zero infrastructure, and is Anthropic's documented recommended embedding partner
for Claude-based retrieval systems.

---

## Why Retrieval Quality Is Everything Here

Every user interaction with this platform starts with one operation: finding the right
API endpoint from a natural language query. If `spec_search` returns the wrong endpoint,
every downstream step — schema validation, diff analysis, self-healing guidance — operates
on the wrong schema. The user gets a confidently wrong answer.

This makes the embedding model the single highest-leverage technical decision in the
entire architecture. A 5% improvement in retrieval accuracy is not a nice-to-have —
it is the difference between Demo 1 working and Demo 1 failing in front of judges.

---

## Why Voyage AI Over the Alternatives

### The benchmark picture

Voyage AI publishes retrieval quality benchmarks using NDCG@10 across eight domains
spanning 100 datasets. Two of those domains — **finance** and **code** — are directly
relevant to our use case. API schema text contains JSON field names, HTTP methods,
enum values, and banking terminology. This is not generic sentence similarity. It is
structured, domain-specific technical text.

The results from Voyage AI's published benchmarks heading into the Voyage 4 launch are
clear:

- `voyage-3-large` outperformed `OpenAI text-embedding-3-large` by **9.74%** on average
  across all domains, and outperformed `Cohere embed-english-v3.0` by **20.71%**
- `voyage-3.5` outperformed `OpenAI text-embedding-3-large` by **8.26%** at the same
  1024-dimension default and at 2.2× lower cost
- The full Voyage 4 series now tops the public **RTEB leaderboard** above Gemini,
  Cohere, and OpenAI

For structured API text specifically, Voyage's domain-specific training emphasis on
finance and code is a direct fit that generic models cannot match.

### Dimension efficiency

`OpenAI text-embedding-3-large` requires 3,072 dimensions to reach its peak quality.
Voyage AI reaches better quality at **1,024 dimensions** — one third the vector size,
one third the storage cost in pgvector, and faster cosine search on our hardware. We
are not sacrificing accuracy for compression. We are getting higher accuracy at lower
cost.

### The Anthropic alignment

Anthropic's own documentation and Cookbook examples reference Voyage AI as the preferred
embedding provider for retrieval pipelines built on Claude. When Claude is the agent
layer and Voyage AI is the retrieval layer, the pipeline has been validated together.
Choosing Voyage AI in a Claude-powered system signals architectural awareness, not just
model familiarity.

### Developer simplicity

The entire integration is a single function call against a managed API. There is no
self-hosted model, no GPU provisioning, no tokenizer configuration, no fine-tuning
pipeline. In a 48-hour solo build, avoiding infrastructure complexity is a legitimate
architectural principle. Voyage AI's Python SDK is stable, minimal, and well-documented.

## Why `voyage-4` Specifically

### What the Voyage 4 series is

Announced January 15, 2026, the Voyage 4 series is a generational upgrade that
introduces two capabilities no previous embedding series has offered:

**Shared embedding space.** All four models in the series — `voyage-4-large`,
`voyage-4`, `voyage-4-lite`, and `voyage-4-nano` — produce compatible embeddings.
Vectors generated by different models in the series can be compared and searched
interchangeably. This is an industry first. For a platform that stores historical
spec versions in pgvector, this matters: any future model upgrade within the Voyage 4
family requires zero re-indexing. The data stays intact.

**Mixture-of-experts architecture.** `voyage-4-large` uses MoE to deliver top-of-
leaderboard accuracy at 40% lower serving cost than comparable dense models. The quality
gains cascade down to the full series through distillation and shared training.

### Why `voyage-4` and not the variants

**Not `voyage-3`**: `voyage-4` is a strict improvement — better accuracy, same price,
same 1024-dimension default, same Python API. There is no reason to use the older model.

**Not `voyage-3.5`**: Solid model that outperforms `OpenAI text-embedding-3-large` by
over 8%, but it predates the Voyage 4 architecture and does not participate in the shared
embedding space. `voyage-4` supersedes it as the recommended general-purpose choice.

**Not `voyage-4-large`**: The highest-accuracy model in the series, best suited for
large-scale document indexing where the upfront embedding cost is small relative to the
ongoing query volume. At our data scale — hundreds of endpoints across a handful of spec
versions — the incremental accuracy gain over `voyage-4` is not observable. `voyage-4`
is the right balance between quality and simplicity for this build.

**Not `voyage-4-lite` or `voyage-4-nano`**: Optimised for query-time latency and local
development respectively. Our constraint is retrieval accuracy at indexing time, not
query latency. Using a lite model here trades accuracy for a speed gain we do not need.

### The Voyage 4 upgrade path

Because of the shared embedding space, future upgrades are non-disruptive:

| Future need | What you do | Re-index required? |
|---|---|---|
| Higher accuracy at query time | Use `voyage-4-large` for queries | No |
| Lower latency at query time | Use `voyage-4-lite` for queries | No |
| Scale to millions of endpoints | Asymmetric: index with large, query with lite | No |
| Domain-specific accuracy | Switch to `voyage-finance-2` | Yes (one-time) |

The architecture does not need to change. The pgvector schema does not need to change.
The 1024-dimension column supports all Voyage 4 variants at their default setting.
